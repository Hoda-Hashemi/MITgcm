This section checks whether each submitted model time step is small enough for the resolved velocities and grid spacing.

Equations

The advective audit uses the local spherical-polar metrics from the 0.25 degree grid:

\[C_{adv}=\Delta t\left(\frac{|u|}{\Delta x}+\frac{|v|}{\Delta y}\right)\]
\[\Delta x=a\cos\phi\,\Delta\lambda,\quad \Delta y=a\Delta\phi\]
\[C_g=\frac{\sqrt{gH}\,\Delta t}{\Delta x}\]

The polar row dominates the metric audit: center-cell dx_min=60.65 m, while dy=27798.73 m. The gravity-wave CFL is shown as an explicit-wave diagnostic only because every active data file has implicitFreeSurface=.TRUE..

How deltaT Was Chosen

The template input/data files provide the grid and default schedule, but the submitted Slurm files are the source of truth for runs. Each job exports DELTA_T, computes nTimeSteps=round(TOTAL_SECONDS/DELTA_T), copies the input directory, then rewrites the run-local data file. The table keeps template, job, and archived-run deltaT separate.

TC1 has the most visible discrepancy: its legacy tools/check_cfl_tc1.py prints the template deltaT=1 s, while the submitted jobs use 60 s, 10 s, or 0.75 s. Use this audit table for submitted-run CFL decisions.

The small near-polar/tilted cases use 0.75 s, the mildly tilted cases use 10 s, standard zonal cases use 60 s, and TC6 uses 30 s.

Decision

No completed run exceeds advective CFL 1.0. TC2 alpha=0.05 is above the conservative 0.5 margin in saved output, so use deltaT<=8.93 s if that margin is required. TC5 becomes non-finite after iteration 1440 despite an initial advective CFL of 0.043, so that case needs a run-health investigation rather than a CFL-only explanation.

TC4 has completed run_u0_20 output, so its saved advective CFL is read from archived U/V fields. TC7 has staged analyzed input, but the preflight audit flags deltaT=60 s as too large before a final rerun.
